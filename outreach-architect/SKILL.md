# Outreach Architect

Один скіл = повний цикл вихідних **LinkedIn**-продажів у діалозі: **ICP → ПЛАН (пошук+валідація+перевірка) → список → валідація через Apify → LinkedIn-цепочка дотиків.**

Канал — тільки LinkedIn. Email-послідовностей у скілі немає.

Це скіл-обгортка. Розумова робота (кого, чому, як писати) — тут. Технічні виклики (`apollo:prospect`, `apollo:verify`) — під капотом, якщо Apollo MCP підключений.

**Залізне правило (з лекції):** Claude вирішує `КОГО` і `ЧОМУ`. Apollo приносить `СПИСОК`. Ти — `ДОТИК`.

---

## Коли запускати

Учасник пройшов L1 (профіль + контекстна модель) і L2 (контент) і тепер хоче вихідний канал — системно знаходити й торкатися ідеальних клієнтів.

**Передумова:** `_context-model.md` (або brief) з L1 — щоб тягнути positioning / offer / claims. Якщо немає — скіл проведе скорочене ICP-інтерв'ю сам, але краще спершу `/company-context-model`.

**Опційно:** Apollo MCP залогінений (`/mcp` → Apollo → connected) — тоді скіл переходить у режим виконання, а не лише поради.

---

## Що питаємо ПЕРЕД стартом (одне повідомлення)

1. **Контекст** — чи є `_context-model.md` / brief? Дай шлях або скажи "немає".
2. **Apollo MCP** — підключений? (`/mcp` → Apollo). Так = режим виконання. Ні = advisory (готові фільтри + копірайт).
3. **Voice** — чи є `voice-profile.json` з L1? Цепочка пишеться у твоєму голосі, якщо так.
4. **Apify** — чи є Apify token / скіл `linkedin-enrich`? Для валідації аудиторії (ФАЗА 6).

Далі — 7 фаз. **Спершу — ФАЗИ 1-4 (інтерв'ю + ПЛАН), і лише після approve плану — виконання (5-7).**

## ФАЗА 1 — ICP-інтерв'ю (задати ВСІ питання, по одному блоку)

Якщо є контекстна модель — спершу показати, що вже відомо (offer, positioning), і питати лише дірки. Якщо ні — пройти всі.

**Блок A — Хто клієнт:**
1. Індустрія / вертикаль (1-3 максимум).
2. Розмір компанії (співробітники: 1-10 / 11-50 / 51-200 / 201+).
3. Гео (країни / міста).
4. Виручка / стадія (bootstrapped / seed / Series A+), якщо релевантно.

**Блок B — Хто саме людина:**
5. Посада ЛПР (хто підписує чек) + хто інфлюенсер (хто рекомендує).
6. Seniority (`c_suite` / `vp` / `director` / `manager`).

**Блок C — Біль і тригер (найважливіше):**
7. Топ-1 біль ICP у його ж словах (verbatim, не маркетингова обгортка).
8. Який buying signal робить його гарячим? (наймає роль / підняв раунд / змінив посаду / постить про проблему / змінює стек).
9. Який стек / інструменти = ознака "наш клієнт"? (напр. HubSpot, Salesforce, Webflow).

**Блок D — Оффер і докази:**
10. Що пропонуємо в 1 реченні + головне заперечення ICP.
11. Який proof point / кейс з цифрою можемо згадати в дотику? (з §10b Case Library, якщо є).

→ Синтез у `outreach/icp.md` (шаблон у `templates/icp-template.md`). Показати, дати approve / edit.

## ФАЗА 2 — Рекомендація підходу (як це робити)

На основі ICP + наявних сигналів сказати ОДНУ річ: з чого почати.

Логіка вибору:
- Є явний hiring-сигнал → почати з intent-пошуку (`organizations_job_postings`).
- Є ідеальний існуючий клієнт → почати з lookalike.
- Сигналу немає, ICP широкий → почати з hard-фільтрів + LLM-верифікації, щоб звузити.
- ICP туманний → спершу витягнути паттерн з історії розмов (схема #10), потім будувати.

Вивести 2-3 речення: "Для твоєї аудиторії раджу почати з [підхід], бо [сигнал]. Це дасть найтеплішу базу."

## ФАЗА 3 — Підбір схем з 37 (рекомендувати, які пасують)

Прочитати `reference/framework-matrix.md` → зіставити ICP-профіль зі схемами → порадити **2-3 (не більше)** з обґрунтуванням.

Формат виводу:
```
Під твою аудиторію [тип] раджу 3 схеми:

⭐ #6 Hiring = buying signal — бо ти таргетиш компанії з болем у [відділ]. Найгарячіший сигнал.
   #12 Один фаундер → схожі — у тебе вже є 2 ідеальні клієнти, зробимо lookalike.
   #24 Apollo дає контакти, Claude дає reasons — щоб кожен дотик був персональний, не шаблон.

Не бери зараз: схеми vibes-категорії (#16-19) — вони для широкого пошуку, тобі треба точність.
```

Завжди пояснювати ЧОМУ саме ці і чому НЕ інші — це навчає студента думати, а не копіювати.

## ФАЗА 4 — ПЛАН (показати ДО пошуку, отримати approve) ⭐

Це і є "перший етап". Перш ніж щось шукати — скіл проговорює весь маршрут трьома рядками і чекає "так". Студент має бачити логіку наперед.

**Формат виводу (приклад):**
```
Ось план. Підтверди — і поїхали:

1️⃣ ПОШУК (Apollo). Шукаємо за фільтрами:
   • person_seniorities: [vp, director]   • posada: Head of Marketing / CMO
   • розмір: 11-50   • гео: DE/NL/UK   • стек: HubSpot
   • hiring-сигнал: organizations_job_postings.posted_titles = ["Marketing Manager"]
   → очікую ~20 лідів.

2️⃣ ВАЛІДАЦІЯ (Apify). Перевіряємо кожного за фільтрами:
   • LinkedIn-активність ≥5 подій / 30 днів (інакше інвайт повисне)
   • актуальність посади (чи людина досі тут)
   • свіжий тригер (пост про біль / найм за 30 днів)
   → actor знайду в Apify Store, покажу ціну до запуску.

3️⃣ ПЕРЕВІРКА (промпт). Прогоню валідованих через ICP-qualifier:
   fit_score (0-10) проти твого ICP + timing hot/warm/cold + reason + наступний дотик.
   → залишаться тільки "гарячі живі". По них пишемо цепочку.

Approve? [так / змінити фільтри]
```

Чекати підтвердження. Тільки після "так" — ФАЗА 5.

## ФАЗА 5 — Пошук аудиторії (Apollo)

**Якщо Apollo MCP підключений (режим виконання):**
- Зібрати погоджені фільтри у виклик `apollo:prospect` → дістати топ-20 лідів з LinkedIn URL.
- Вивести табличку: Name | Title | Company | LinkedIn | Hiring Signal.

**Якщо Apollo MCP НЕ підключений (advisory):**
- Видати ГОТОВІ Apollo-фільтри під вставку (JSON-блок).
- Hard-rule нагадати: `organizations_job_postings.posted_titles` (наймають = intent), НЕ `q_organization_job_titles` (роль вже є).

## ФАЗА 6 — Валідація аудиторії через Apify (LinkedIn)

Прочитати `reference/apify-validation.md`. Apollo дав список — але дані бувають застарілі. Перш ніж писати цепочку, провалідувати по 3 LinkedIn-вимірах: **активність · актуальність посади · свіжий тригер.**

**Логіка скіла:**
1. Пояснити, ЧОМУ валідуємо (Apollo staleness — табличка ризиків з reference).
2. Для outbound критично — LinkedIn-активність (active = response rate ×3-4).
3. **Піти в Apify Store і знайти actor** під вимір:
   ```bash
   curl -s "https://api.apify.com/v2/store?search=linkedin+profile+posts&limit=5" \
     | jq '.data.items[] | {name, username, runs: .stats.totalRuns, price: .currentPricingInfo.pricePerUnitUsd}'
   ```
   → показати 2-3 топ-actors за `totalRuns` + ціну.
4. **Запропонувати запустити** (показати ціну ДО запуску, дочекатись "так"):
   - спершу прочитати input schema: `GET /v2/acts/{ID}/input-schema` — не вгадувати поля;
   - запуск: `POST /v2/acts/{ID}/runs?token=$APIFY_TOKEN` (input = LinkedIn profile URLs);
   - результат: `GET /v2/acts/{ID}/runs/last/dataset/items`.
5. **Fallback:** якщо є скіл `linkedin-enrich` — запропонувати його замість ручних curl (обгортає весь пайплайн).

→ Оновити `outreach/leads.md`: колонки `active` · `still_at_company` · `trigger`. Відсіяти мертвих у `parked`. Сказати цифру: з 20 → зазвичай 12-15 живих.

**⚠️ Не запускати жоден actor без явного "так" студента і без показаної ціни.**

## ФАЗА 7 — Цепочка LinkedIn-дотиків (Touchpoint Loop)

Пишемо цепочку ТІЛЬКИ по валідованих ("живих") лідах із ФАЗИ 6. Канал — тільки LinkedIn.

Прочитати `reference/sequence-playbook.md`. Згенерувати повну цепочку під сегмент, у голосі студента (voice-profile, якщо є).

**Default = Tier 1 (Claude драфтить, людина відправляє вручну — безпечно, як домовлено в L2):**

```
Day 0:  LinkedIn invite (≤200 знаків, 1 конкретна деталь з профілю, гачок-питання, БЕЗ "love your content")
Day 3:  ЯКЩО не прийняв → лайкнути 2 останні пости (Claude каже які)
Day 5:  ЯКЩО прийняв → DM з кейсом під ЇХНІЙ контекст (proof point, НЕ продаж, м'яке питання в кінці)
Day 7:  ЯКЩО прийняв, не відповів → коментар під свіжим постом (value-add, не "great post")
Day 12: ЯКЩО тиша → другий DM (інший кут, нова цінність — НЕ "просто нагадую")
Day 18: ЯКЩО тиша → фінальний м'який DM ("закриваю тему, якщо актуально — ось як") → СТОП
```

Для кожного кроку згенерувати РЕАЛЬНИЙ текст під сегмент + proof point з ФАЗИ 1. Не плейсхолдери.

**Tier 2 (advanced, на запит) — нагадати про ризик банy:**
- Авто-відправка інвайтів / DM через Unipile / webhook (Apollo `invite_accepted` → n8n → DM).
- Ліміт 80-100 інвайтів/тиждень + warm-up.
- Сказати прямо: "Це advanced і ризикує акаунтом. Default — Tier 1."

→ Зберегти у `outreach/sequences/[segment-slug].md` (шаблон у `templates/sequence-template.md`).

---

## Quality-rules для цепочки (обов'язково)

- ❌ Em-dash (—), tier-1 AI-слова (leverage, delve, robust), "great post", compliment sandwich, "love your content".
- ✅ 1 конкретна деталь з профілю/компанії в інвайті.
- ✅ Proof point з цифрою у DM.
- ✅ Кожен крок має власну цінність (не "просто нагадування").
- ✅ Інвайт ≤200 знаків; DM ≤150 слів; другий DM ≤100 слів.
- ✅ Тільки LinkedIn-дії (invite / like / comment / DM). Жодного email.
- Фінальний DM без відповіді → СТОП. Не спамити.
